# Architectural Decisions

## [DECISION-001] MCP-first delivery over file injection
- **Date:** 2026-01-15
- **Status:** Accepted
- **Decision:** Context is delivered via MCP tool results, not by writing to CLAUDE.md or injecting into the system prompt prefix.
- **Reason:** CLAUDE.md loads at the front of the prompt cache prefix. Every task change triggers a full prefix re-write at 1.25× the base input price. MCP tool results land in the conversation tail — no prefix write per task. On a 10K-token context layer this saves ~$0.08–$0.14 per session.
- **Alternatives considered:** Writing full context to CLAUDE.md (original v0.1 approach); pointer-only CLAUDE.md with full content in a sidecar file.

## [DECISION-002] `.ai-context/` as canonical directory (`.cram-ai-context/` legacy fallback)
- **Date:** 2026-02-01
- **Status:** Accepted
- **Decision:** New repos use `.ai-context/` as the context directory. The old `.cram-ai-context/` path is silently supported via fallback in `context_dir.py`.
- **Reason:** Shorter, tool-agnostic name. The context layer is not cram-specific — it should feel like a repo artifact, not a vendor directory.
- **Alternatives considered:** Keep `.cram-ai-context/` permanently; rename with a migration script.

## [DECISION-003] Task slot namespacing per `get_context()` call
- **Date:** 2026-02-10
- **Status:** Accepted
- **Decision:** Each `get_context("task")` call writes to `.ai-context/tasks/<slug>.md` instead of a shared `CURRENT_TASK.md`.
- **Reason:** Multiple agents running concurrently on the same repo would overwrite each other's task context when sharing a single file. Per-task slots eliminate the contention without coordination.
- **Alternatives considered:** Shared CURRENT_TASK.md (original approach — caused multi-agent clobbering); lock files; per-PID files.

## [DECISION-004] Cheap model for context generation, not the session model
- **Date:** 2026-01-20
- **Status:** Accepted
- **Decision:** ARCHITECTURE.md and SYMBOLS.md are generated by a cheap/fast model (Haiku or equivalent), not the full session model.
- **Reason:** Context generation runs on every `cram sync` (post-commit hook). Cost and latency must be negligible. The output is a summary, not creative work — quality gap is acceptable.
- **Alternatives considered:** Use the same model as the coding session; use static analysis only (no LLM).

## [DECISION-005] Output byte-cap protection by default
- **Date:** 2026-03-01
- **Status:** Accepted
- **Decision:** All shell commands run by agents via cram context must be byte-capped (`| head -c 6000`). This is documented in the context layer delivered to every agent.
- **Reason:** Unbounded command output fills the context window with noise. A single `cat` on a large file or `git log` without limits can consume thousands of tokens in one tool call. The cap is injected as a rule into CURRENT_TASK.md so it applies without any per-session configuration.
- **Alternatives considered:** Trust agents to self-limit; truncate at the MCP layer; token counting middleware.

## [DECISION-006] Git-commit-count staleness score over mtime
- **Date:** 2026-03-15
- **Status:** Accepted
- **Decision:** Context health is measured as commits-since-last-sync, not wall-clock time since last file modification.
- **Reason:** mtime is unreliable — `git pull` updates mtime even when ARCHITECTURE.md wasn't regenerated. Commit count is a direct proxy for "how much has the codebase changed since context was last updated."
- **Alternatives considered:** mtime delta (original approach — kept as fallback when git unavailable); line-diff percentage.

## [DECISION-007] Decisions need human review before becoming permanent record
- **Date:** 2026-04-01
- **Status:** Accepted
- **Decision:** Agent-proposed decisions (via `propose_decision` MCP tool) are written with a `[PENDING]` tag and "Pending — proposed by agent, awaiting owner review" status. They are not treated as accepted until a human approves them in `cram ui` or edits the file directly.
- **Reason:** Agents can misidentify a local implementation choice as a project-wide architectural decision. The trust boundary must sit with the human owner, not the agent.
- **Alternatives considered:** Auto-accept all agent proposals; require separate approval file; ignore agent write-back entirely.

## [DECISION-008] litellm as the model abstraction layer
- **Date:** 2026-01-10
- **Status:** Accepted
- **Decision:** All model calls go through litellm, which translates provider-specific APIs into a unified interface.
- **Reason:** cram needs to work with OpenAI, Anthropic, Google, Ollama, and enterprise proxies. Maintaining per-provider adapters is not the core product. litellm handles auth, retries, and the OpenAI-compatible wire format.
- **Alternatives considered:** Direct Anthropic SDK only; httpx with manual provider routing; langchain.

## [DECISION-009] Textual for TUI — optional dependency
- **Date:** 2026-04-15
- **Status:** Accepted
- **Decision:** `cram ui` requires `pip install cram-ai[tui]` — Textual is not a default dependency.
- **Reason:** Textual adds ~15MB to the install. Most users will never use the TUI (they use the MCP path). Making it optional keeps the default install lightweight.
- **Alternatives considered:** Always install Textual; use curses directly; no TUI (CLI-only).
